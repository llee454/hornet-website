Definition: What is Formally Verified Software Development?

Formally verified software development is the practice of writing mathematical proofs to verify that software is correct.

A mathematical proof is a sequence of assertions in which every assertion is either an axiom or can be derived from one or more of the preceding assertions by applying a logical deduction rule.

I specialize in writing proofs in Coq, a programming language capable of expressing both logical arguments and describing algorithms.

Industrial Results

In 2018, SiFive, a Silicon Valley-based company specializing in the design of RISC-V processors used an algorithm that they feared was incorrect.

The floating point unit used within their S-series processors relied on an algorithm that they feared was incorrect. The engineer tasked with verifying the algorithm had stalled while trying to prove that the registers allocated by the unit where large enough to store the intermediate values computed by the unit during square root and division operations. While the unit passed SiFive's test suite, SiFive feared that a bug lurked in an edge case not covered by their tests.

I successfully derived an analytical proof verifying that the registers allocated by the unit where sufficiently large to store these intermediate values. To do this, I mapped the algorithm used by the unit onto a recursive algorithm that could be described by a system of recurrence relations. I then proved that the maximum value computed by each iteration of this simpler algorithm was less than the maximum value that could be stored in the allocated registers.

Introduction

SiFive had developed the unit in a modeling language named Chisel. I began by writing a set of recurrence relations that formalized the square root and division algorithms. I then identified a bijective mappings between the variables used in SiFive's model and the variables used in my equations. This allowed me to prove that my equations correctly modeled the unit. I then verified that my equations correctly converged onto the correct results, thereby verifying that SiFive's unit was logically correct.

One problem remained however. SiFive's unit defined a register named rem. SiFive feared that the rem register was too small to store all of the intermediate values computed during operation. To prove that the rem register was large enough, I needed to prove that the values stored within rem were smaller than 2n, where n represented the number of bits SiFive had allocated.

This register corresponded to a variable defined within my equations named error(n). Using the mapping between rem and error(n), I had to verify that: $$\forall n, error(n) < \frac{8}{2^n}$$.

Portfolio: Large Scale Developments Using Coq

While working with SiFive, I was the lead developer for the Kami Processor, an Open-Source Risc-v processor written in Kami.

Kami is a domain-specific language designed for modeling digital circuits.

The Kami processor is a highly configurable RISC-V processor that implements a substantial fraction of the RISC-V feature set: including memory virtualization, compressed instruction encodings, floating point operations, and dynamic data widths.

Kami is embedded within Gallina, Coq's dependently-typed programming language. Using Coq, I've verified components used within the Kami Processor and verified the Gallina functions used to generate the Kami Processor model.
